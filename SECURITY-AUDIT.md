# Аудит безопасности сайта НПП «КПК»

Проведён анализ кода на типичные уязвимости. Ниже — находки и рекомендации.

---

## 1. Риск XSS (Cross-Site Scripting)

### 1.1 Галерея на главной (`script.js`)

- **Где:** при открытии фото в галерее подпись выводится через `innerHTML`.
- **Риск:** если заголовки карточек (`h3`) когда-нибудь будут подставляться из CMS или пользовательского ввода, возможен запуск скриптов в браузере посетителя.
- **Исправление:** для вывода подписи использовать `textContent` вместо `innerHTML` (уже внесено в `script.js`).

### 1.2 Модальное окно «Подробнее» в каталоге (много страниц)

- **Где:** в страницах каталога, новостей, услуг скрипт выводит в модалку:
  - `propsEl.innerHTML = this.dataset.itemProps || ''`
  - `descEl.innerHTML = this.dataset.itemDesc || 'Нет описания'`
- **Риск:** если атрибуты `data-item-props` или `data-item-desc` будут заполняться из базы/CMS (например, описание товара), злоумышленник или ошибка в данных могут подставить HTML/JS, и произойдёт XSS.
- **Рекомендация:**
  - если описание и свойства — только текст: везде заменить `innerHTML` на `textContent`;
  - если нужна разметка (жирный, списки): использовать библиотеку санитизации HTML (например, DOMPurify) перед присвоением в `innerHTML`.

---

## 2. Формы обратной связи

- **Наблюдение:** формы (`#modal-form`, `.callback-form`) не имеют атрибутов `action` и `method` в разметке.
- **Следствие:** отправка, скорее всего, обрабатывается через JavaScript или не обрабатывается (статический сайт). В репозитории серверной части не видно.
- **Риск:** при появлении бэкенда важно:
  - не подставлять ввод пользователя в HTML без экранирования/санитизации;
  - не собирать SQL-запросы из ввода пользователя (риск SQL-инъекций);
  - проверять и ограничивать ввод (длина, тип данных).

---

## 3. Внешние ресурсы (CDN)

- Подключаются: jQuery, Remodal, Google Fonts, изображения с внешних доменов (z-km.com, siteapi.org, pantes.ru).
- **Риск:** при компрометации CDN или подмене ресурса возможна поставка вредоносного скрипта/стилей (supply chain). Снижение риска: по возможности подключать критические скрипты со своего домена и проверять целостность (SRI).

---

## 4. Прочее

- **Контакты:** телефон и email в открытом виде в HTML — нормально для визитки; для защиты от спама можно усложнить вывод (например, собирать строку в JS), но это не «взлом» в классическом смысле.
- **Сайт статический:** нет доступа к БД и серверной логике из фронтенда, поэтому классические SQL-инъекции по текущему коду невозможны.

---

## Итог

Реальных «взломов» по текущему коду не обнаружено. Основные рекомендации:

1. Не использовать `innerHTML` для данных из БД/CMS без санитизации; где достаточно текста — использовать `textContent`.
2. При появлении бэкенда для форм — валидация и экранирование на сервере, подготовленные запросы к БД.
3. По возможности подключать критические скрипты со своего сервера и использовать SRI для CDN.
